## TL;DR — где мы

**Готово к демо клиенту:** клиентский сайт, вся админка (пакеты + автомобили +
бронирования v2 + Telegram-заявки), 3-шаговый booking с филиалом
и слотами, редактирование записи, маска телефона, toast-уведомления,
skeleton-лоадеры, mobile bottom-bar, error boundary, рекомендация замены
масла по пробегу, загрузка фото пакета. Можно проходить весь основной
сценарий: регистрация → добавление авто → выбор пакета → запись с
филиалом/датой/временем/пробегом → просмотр в сервисной книжке →
редактирование → админ видит запись в `/admin/bookings`.

**Заблокировано бэком (~10% функционала):** отмена записи (клиентский
PATCH/cancel уже работает, нужна только серверная серверная сортировка
bookings и пара мелочей), восстановление пароля (UI готов), профиль
клиента, отзывы, корректные картинки логотипов и авто из S3.

**Заблокировано ПМом/дизайнером:** макет профиля, макет отзывов.

**Мобильное приложение:** делается отдельным нативным приложением
(Expo / React Native, репозиторий `sct-mobile`) — клиентская часть веба
без админки. Не WebView. Статус и паритет с вебом — в
`../sct-mobile/PORTING_STATUS.md`.

**Финансовый прогноз:** при текущем темпе и если бэк закроет блокировки в
ближайшие 1–2 недели, MVP под сдачу готов через 2–3 рабочих дня моей
работы.

## 4. Технические известные баги и костыли

Не блокирует работу, но при первой возможности фиксим:

1. **Главный Hero — декоративный силуэт вместо фото мастера.**
   Когда заказчик пришлёт брендовое фото — заменим в одном файле
   `features/home/HomeHero.tsx`.
2. **Логотипы марок и фото авто — серые инициалы.**
   Связано с S3 403. Когда бакет откроют — картинки подгрузятся
   автоматически.
3. **Слоты времени строятся фронтом** в `features/booking-wizard/lib.ts`
   (шаг 30 минут от open до close). Когда бэк добавит
   `/slots/?date=&service_station_id=&duration_min=` — заменим генератор
   на запрос к API.
4. **Промо-баннер на главной** показывает таймер до конца месяца —
   хардкод. Когда появится сущность `Promo` с `end_date` — заменим.
5. **`page_size` на `/admin/packages`** принимает только {10, 20, 50, 100}
   — иначе 400. Селект уже ограничен.
6. **Status `BLOCKED` у клиента** — фронт пускает в ЛК если есть валидный
   токен. Лучше бэк возвращал бы 403 на `/auth/profile/` для BLOCKED.
7. **Поле «Промокод» при регистрации** отправляется на бэк, но в
   `ClientRegisterRequest` не описано. Когда ПМ решит — либо подключат на
   бэке, либо уберём поле.
8. **Клиентская сортировка bookings по «Времени»** — костыль до фикса
   `ordering` на бэке (см. §3.1).

## 6. Архитектурные решения

Полезно знать, если кто-то будет ревьюить или подключаться:

- **Два независимых axios-инстанса** (`http` для клиента, `staffHttp`
  для админа) — чтобы 401 в админке не сбрасывал клиентскую сессию.
- **Модалки авторизации глобально через `AuthModalsHost`** в `Layout`, а
  не на каждой странице. Работают через `?modal=login|register|forgot-password`.
- **«Сервисная книжка» и «Главная» используют общую правую колонку**
  `MyGarageColumn`. Изменения в гараже отражаются мгновенно через
  TanStack Query invalidations.
- **Wizard записи на сервис не использует роутинг для шагов** — state
  machine внутри одного компонента. Упрощает state, но back-кнопка
  браузера выходит со страницы. Если ПМ захочет — переделаем.
- **Состав пакета в админ-форме** — динамический список через
  `useFieldArray` (RHF), fuzzy-search через
  `/staff_endpoints/packages/package-items/search/?q=&limit=`.
- **Картинки от бэка везде через `SafeImage`** — при 403/404 показывает
  initially-генерируемый фолбэк. Критично, потому что S3 у бэка закрыт.
- **Загрузка фото пакета — двухшаговая:** сначала JSON-PATCH со всеми
  полями (включая `package_items[]`), потом отдельный multipart-PATCH
  только с `image`. Чтобы не упаковывать массивы в multipart.
