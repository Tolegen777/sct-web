## TL;DR — где мы

**Готово к демо клиенту:** клиентский сайт, **вся админка** (пакеты + автомобили),
полный 4-шаговый booking с филиалом и слотами, редактирование записи,
маска телефона, toast-уведомления, skeleton-лоадеры, mobile bottom-bar,
error boundary. Можно проходить весь основной сценарий:
регистрация → добавление авто → выбор пакета → запись с филиалом, датой,
временем, пробегом → просмотр в сервисной книжке → редактирование.

**Заблокировано бэком (~15% функционала):** отмена записи, восстановление
пароля (UI готов), профиль клиента, отзывы, корректные картинки логотипов
и авто.

**Заблокировано ПМом/дизайнером:** макет профиля, макет отзывов, решение
по WebView vs нативный React Native для мобильного приложения.

**Финансовый прогноз:** при текущем темпе и если бэк закроет
блокировки в ближайшие 1–2 недели, MVP под сдачу готов через
2–3 рабочих дня моей работы.

## 4. Технические известные баги и костыли

Не блокирует работу, но при первой возможности фиксим:

1. **Главный Hero — декоративный силуэт вместо фото мастера.**
   Когда заказчик пришлёт брендовое фото в высоком разрешении —
   заменим в одном файле `features/home/HomeHero.tsx`.

2. **Логотипы марок и фото авто — серые инициалы.**
   Связано с S3 403 (см. 3.1). Когда бакет откроют — картинки
   подгрузятся автоматически.

3. **Слоты времени строятся фронтом** в `features/booking-wizard/lib.ts`
   (шаг 30 минут от open до close). Когда бэк добавит
   `/slots/?date=&service_station_id=&duration_min=` —
   заменим генератор на запрос к API, чтобы видеть реально занятые
   слоты.

4. **Промо-баннер на главной** показывает таймер до конца текущего
   месяца. Это хардкод. Когда появится сущность `Promo` с `end_date` —
   заменим на реальный.

5. **Адресная строка фильтров на `/admin/packages` принимает page_size**
   только из {10, 20, 50, 100} — иначе бэк отвечает 400. Селект на
   фронте уже ограничен этими значениями.

6. **Status `BLOCKED` у клиента** — фронт пускает в ЛК если есть валидный
   токен. Лучше бэк возвращал бы 403 на `/auth/profile/` для
   `BLOCKED`-юзеров. Сейчас работает, но защита неполная.

7. **Поле «Промокод» при регистрации** отправляется на бэк, но бэк его
   не описывает в `ClientRegisterRequest`. Не падает (DRF молча
   игнорит лишнее поле в `extra_kwargs={'allow_unknown': True}` —
   зависит от настроек). Когда ПМ решит, использовать ли промокоды —
   либо подключат на бэке, либо уберём поле.

## 6. Архитектурные решения, которые приняли по дороге

Полезно знать, если кто-то будет ревьюить или подключаться:

- **Два независимых axios-инстанса** (`http` для клиента, `staffHttp`
  для админа) — чтобы 401 в админке не сбрасывал клиентскую сессию.
- **Модалки авторизации глобально через `AuthModalsHost` в `Layout`**, а не
  на каждой странице. Работают через `?modal=login|register|forgot-password`.
- **«Сервисная книжка» и «Главная» используют общую правую колонку**
  `MyGarageColumn`. Изменения в гараже отражаются мгновенно через
  TanStack Query invalidations.
- **Wizard записи на сервис не использует роутинг для шагов** —
  state machine внутри одного компонента. Это упрощает state, но
  back-кнопка браузера выходит со страницы. Если ПМ захочет —
  переделаем на роутинг.
- **Состав пакета в админ-форме** — динамический список через
  `useFieldArray` (RHF), автокомплит через `/staff_endpoints/packages/package-items/?autocomplete=1&q=...`.
- **Картинки от бэка везде через `SafeImage`** — при 403/404 показывает
  inicially-генерируемый фолбэк. Это критично, потому что S3-бакет
  у бэка пока закрыт.
