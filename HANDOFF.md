## Что я делаю по умолчанию (правила)

- **Никаких новых зависимостей без спроса.** Toast, Skeleton, ErrorBoundary
  — всё своё, 200 строк кода вместо 30 КБ npm-пакетов.
- **`as unknown as`-касты допустимы только** там, где OpenAPI бэка не
  отражает реальный ответ. Каждое такое место документировано в
  BACKEND_NOTES §1.
- **Сначала проверяю реальный ответ API** через `curl` или
  `additional/full_api_healthcheck.py`, потом подключаю на фронте. Не
  доверяю одной только OpenAPI schema — у этого бэка много расхождений.
- **TaskCreate/TaskUpdate** для многошаговых задач. Текущий список —
  через TaskList.
- **Build после каждого изменения**: `npm run build 2>&1 | grep -E
  "error|✓ built" | tail -5`. Build должен быть чистым.
- **Коммиты только когда пользователь явно попросит** (через команду
  «закоммить»).
- **Никаких эмодзи в коде**, если пользователь не попросит.

## Стиль общения с пользователем

- Пользователь — фронт-разработчик с опытом, говорить на ты.
- Любит **подробные ответы с обоснованиями**, не сухие.
- Любит **планы и таблицы** — «что делаем», «что блокирует», «кто разблокирует».
- Когда задача большая — **разбить на партии** и спросить подтверждения,
  прежде чем делать всё сразу.
- Когда нашёл проблему — сначала **диагностируй и объясни**, потом фикси.
- На вопросы «что осталось делать?» — **давай шорт-лист** с приоритетами
  и оценками времени.
- У пользователя есть свой аккаунт `+77075001639` с собственным паролем
  (регистрировал сам, не путать с демо-учётками из README).

## Чего точно НЕ делать

- ❌ Не предлагать переписать `BACKEND_NOTES.md` или удалить из него
  пункты — это рабочий документ для бэкендщика, удалить может только
  бэкендщик (отметив «сделано»).
- ❌ Не коммитить без явной просьбы пользователя.
- ❌ Не вводить новые зависимости (sonner, react-toastify, framer-motion и
  т. д.) без обсуждения.
- ❌ Не убирать `enabled: isAuthed` из query-хуков — это сделано осознанно,
  чтобы гость не дёргал 401.
- ❌ Не править `additional/*` — это исходники от бэкендщика, читать
  только.
